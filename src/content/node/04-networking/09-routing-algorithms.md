## 路由器——互联网的"指路人"

数据包从你的电脑到 Google 的服务器，中间要经过十几台路由器。每台路由器都要**决定下一步往哪走**。

> 🏫 **类比：高速公路导航**
> - 你要从北京到上海（你的电脑到 Google）
> - 每到一个服务区（路由器），导航告诉你：下一个出口往哪个方向
> - 每个服务区不需要知道完整路线——只需要知道"走哪条路离上海更近"

## 距离向量算法（Distance Vector）

每个路由器维护一张表，记录到所有目标网络的"距离"和"下一跳"。

```
路由器 A 的路由表：
┌────────────┬────────┬────────┐
│ 目标网络    │ 距离    │ 下一跳  │
├────────────┼────────┼────────┤
│ 10.0.0.0/8│ 0      │ 直接    │
│ 172.16.0.0│ 2      │ B      │
│ 192.168.0.0│ 3      │ B      │
└────────────┴────────┴────────┘
```

**工作原理（Bellman-Ford 算法）：**
1. 每个路由器定期把自己的路由表发给邻居
2. 邻居收到后：如果通过你能更短到达某网络 → 更新自己的路由表
3. 直到所有路由器收敛（"好消息传得快，坏消息传得慢"）

> ⚠️ **计数到无穷**——距离向量算法的经典问题。当某网络断开，路由器之间会反复交换错误信息，距离数逐渐增大到最大值才停下。

**代表协议：RIP（Routing Information Protocol）**
- 最大跳数 15（超过 15 认为不可达）
- 每 30 秒广播一次路由表
- 适用于小型网络

## 链路状态算法（Link State）

每个路由器了解整个网络的拓扑结构——像有一张完整的地图。

```
链路状态算法步骤：
1. 发现邻居——每个路由器找到直连的邻居
2. 测量距离——到每个邻居的"代价"
3. 构建 LSP（链路状态包）——描述本地连接情况
4. 泛洪 LSP——把自己的 LSP 发给所有路由器
5. 每台路由器都有完整的网络拓扑图
6. 用 Dijkstra 算法计算最短路径树
```

> 💡 链路状态比距离向量收敛更快，但需要更多 CPU 和内存资源。

**代表协议：OSPF（Open Shortest Path First）**
- 基于 Dijkstra 算法（SPF）
- 支持分层（区域）
- 收敛速度快

## 路由协议对比

| 协议 | 算法 | 度量 | 收敛 | 规模 |
|:----:|:----:|:----:|:----:|:----:|
| **RIP** | 距离向量 | 跳数 | 慢（分钟级） | 小（15跳） |
| **OSPF** | 链路状态 | 带宽/延迟 | 快（秒级） | 大 |
| **IS-IS** | 链路状态 | 带宽/延迟 | 快 | 大（ISP骨干） |
| **BGP** | 路径向量 | 多种策略 | 较慢 | 全球互联网 |

## BGP——互联网的"大动脉"

**BGP（Border Gateway Protocol）** 是互联网的核心路由协议——连接不同 AS（自治系统）之间的路由。

```
AS 100（中国电信）─── AS 200（中国联通）
          │                  │
          └── AS 300（腾讯）──┘
```

BGP 不是找最短路径——而是根据**策略**选择路径（商业关系、政治因素）：

| 关系 | 策略 |
|------|------|
| **客户 → 提供商** | 付钱买带宽，可以到达所有互联网 |
| **对等互联** | 免费互通，只交换彼此的客户路由 |

## 小结

| 概念 | 要点 |
|------|------|
| **距离向量** | 邻居之间交换路由表（RIP） |
| **链路状态** | 每台路由器有全网拓扑图（OSPF） |
| **Dijkstra** | 链路状态算法计算最短路径 |
| **BGP** | 互联网核心路由协议，基于策略 |
| **收敛** | 网络变化后所有路由器达成一致的过程 |
