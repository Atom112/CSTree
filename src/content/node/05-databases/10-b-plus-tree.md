## 为什么不用二叉搜索树？

二叉搜索树每个节点只存一个键——在磁盘上每个节点读一个块（4KB），只用一个键太浪费了。B+ 树每个节点存几百个键——一次 I/O 读取一个节点，获取更多信息。

> 🏫 **类比：图书馆书柜**
> B+ 树 = 每层书柜上贴着"A-K"、"L-Z"的标签（内部节点），你走到对应区域（叶子节点）才能看到具体的书。找书只需要看路标，不用遍历每个书架。

```
B+ 树结构：
        [50, 100]
       /    |    \
   [20,35] [70,85] [120,150]
   /   |    |   |    |    \
 数据 数据 数据 数据 数据 数据
```

| 特点 | 说明 |
|------|------|
| **多路** | 每个节点存储上百个键 |
| **平衡** | 所有叶子在同一深度 |
| **有序** | 叶子节点用链表连接，支持范围查询 |
| **磁盘友好** | 节点大小 = 磁盘块大小 |

## 小结

| 概念 | 要点 |
|------|------|
| **B+ 树** | 多路平衡查找树，叶子存数据 |
| **扇出** | 每个节点的子节点数（通常几百） |
| **高度** | 几百万数据只需 3-4 层 |

**为什么先学这个？** B+ 树是 MySQL InnoDB 的默认索引。另一种常见索引是 [[hash-index|哈希索引]]。
